## Purpose

A generated application is the artefact the platform produces for an operator. It is not a platform component — it is a deployable unit that the platform builds, deploys, and monitors but does not own at the application-logic level. Each app is scoped to one operator, lives in their private Gitea organisation, and runs as one or more Docker containers on the platform network.

This document defines the generated app as a runtime entity: its internal structure, resource model, mandatory interface contract, lifecycle states, and the platform components that interact with it. It is the single reference for anyone working on the build pipeline, CI templates, or infra provisioning who needs to know what the pipeline must produce.

## C4 position

Generated apps are modelled as a separate software system at C4 L1 ("Generated Application") and as a container group at C4 L2. They are not containers inside the Foundry system because they are independent deployable artefacts with their own Git repos, CI pipelines, and lifecycles.

Platform components that interact with them are shown in `docs/architecture/structurizr/workspace.dsl`.

## The five app types

Each app type has a fixed structural model. The planner selects the type and chooses the API stack, database type, and whether a frontend is required (from `standards/templates.yaml`). The builder generates code by augmenting the chosen template. The CI pipeline builds and deploys the resulting containers.

| Type | API container | Frontend container | DB | Infra agent called |
|------|--------------|-------------------|-----|-------------------|
| `form` | FastAPI (`fastapi-base`) | Next.js (`mui-minimal`) | SQLite or tenant Postgres | Only if Postgres |
| `dashboard` | FastAPI or Express | Next.js (`mui-minimal`) | None / external read-only | No |
| `workflow` | FastAPI (`fastapi-base`) | Next.js (`mui-minimal`) | Tenant Postgres | Yes |
| `connector` | FastAPI or Express | None | Tenant Postgres or tenant MongoDB | Yes |
| `assistant` | FastAPI (`fastapi-base`) | Next.js (`mui-minimal`, chat layout) | Tenant Postgres (pgvector extension) | Yes |

### `form`

Two containers: FastAPI API + Next.js frontend (MUI Minimal). Browser-facing CRUD — data entry forms with persistence. Default DB is SQLite (embedded, data lost on container removal without a volume); planner upgrades to tenant Postgres when the spec requires durable persistence. No infra agent call for SQLite; infra agent creates a per-app database in the tenant Postgres container for the Postgres variant. Frontend is generated by the ui-designer from the MUI Minimal template.

### `dashboard`

Two containers: FastAPI or Express API + Next.js frontend (MUI Minimal). Read-only data visualisation. Data comes from external sources injected as env vars (read-only DSNs, API keys). No write paths. No infra agent call required.

### `workflow`

Two containers: FastAPI API + Next.js frontend (MUI Minimal). Stateful multi-stage task tracker. Tasks progress through operator-defined stages and require durable persistence — planner always selects tenant Postgres. Infra agent creates the per-app database in the tenant Postgres container before the builder runs.

### `connector`

One container: FastAPI or Express API. Headless backend integration — API proxy, webhook consumer, or data sync job. No browser-facing frontend unless the spec explicitly requests a status page (in which case a second Next.js container is added). Planner selects tenant Postgres for relational integrations or tenant MongoDB for document-oriented integrations. Infra agent provisions the per-app database within the appropriate tenant container.

### `assistant`

Three containers: FastAPI API + Next.js frontend (MUI Minimal, chat-adapted layout) + tenant Postgres with pgvector extension. Conversational UI backed by a RAG pipeline. Operator uploads documents; the app embeds them using pgvector and retrieves them to augment LLM responses. The app calls Ollama directly at runtime for inference. The infra agent ensures the tenant Postgres container is running and has the pgvector extension enabled, then creates a per-app database within it. There is no longer a separate pgvector container — pgvector runs as an extension on the tenant Postgres container (ADR-0004).

## Runtime artefacts

### App container (always present)

The primary Docker container running the generated application.

**Mandatory files produced by the build pipeline:**

Apps with a frontend (form, dashboard, workflow, assistant):
```
{app-name}/
  api/                  # API container (from template + LLM augmentation)
    main.py / index.js
    requirements.txt / package.json
    Dockerfile
  frontend/             # Frontend container (from mui-minimal template + ui-designer)
    src/
    package.json
    Dockerfile
  .woodpecker.yml       # Builds and deploys both containers; generated by builder
  README.md             # Generated by builder docs.py
  docs/api.md           # connector and assistant types only
  docs/data-ingestion.md  # assistant type only
```

Apps without a frontend (connector):
```
{app-name}/
  main.py / index.js    # API files at root
  requirements.txt / package.json
  Dockerfile
  .woodpecker.yml
  README.md
  docs/api.md
```

**Traefik routing for two-container apps:**
- API container: `{name}.APP_DOMAIN`
- Frontend container: `{name}-ui.APP_DOMAIN`

**Docker labels (applied by Woodpecker deploy step):**

| Label | Value | Purpose |
|-------|-------|---------|
| `traefik.enable` | `true` | Opt the container into Traefik routing |
| `traefik.http.routers.{name}.rule` | `Host('{name}.APP_DOMAIN')` | Route hostname to this container |
| `traefik.http.services.{name}.loadbalancer.server.port` | App port (default 8000) | Port Traefik forwards to |
| `platform.managed` | `true` | Marks as platform-managed (from `security.yaml`) |
| `platform.tenant` | `{tenant_id}` | Tenancy marker; used by watchdog to scope container sweeps |
| `platform.repo` | Gitea repo URL | Traceability back to source (from `security.yaml`) |
| `platform.version` | Git short SHA | Image version at deploy time |

**Network:** all generated app containers join `platform_platform` Docker bridge network. This gives them hostname-addressable access to their provisioned resources (`{app-name}-pgvector`, shared Postgres) and to platform services.

### Tenant Postgres container (conditional: `workflow`, `connector`, `assistant`, and `form` with Postgres)

A dedicated Postgres container provisioned once per tenant, shared by all apps within the tenant that require relational storage. See ADR-0004 and `infra.md` for the full lifecycle.

- Container name: `tenant-{tenant_id}-postgres`
- Image: `pgvector/pgvector` (Postgres with pgvector extension pre-installed)
- Joined to `platform_platform` network; accessible at `tenant-{tenant_id}-postgres:5432`
- pgvector extension enabled at the container level; all `assistant` apps in the tenant share this container for vector operations
- **Per-app:** database `{app-name}` and user `{app-name}-user` created by infra agent; `DATABASE_URL` injected via `.env.platform`
- **Per-app teardown:** database and user dropped; container is not removed (shared by other apps)
- **Tenant teardown:** container stopped and removed when all apps in the tenant are deleted

### Tenant MongoDB container (conditional: `connector` with document storage)

A dedicated MongoDB container provisioned once per tenant, shared by all apps within the tenant that require document storage.

- Container name: `tenant-{tenant_id}-mongo`
- Image: `mongo` (pinned version)
- Joined to `platform_platform` network; accessible at `tenant-{tenant_id}-mongo:27017`
- **Per-app:** database `{app-name}` and scoped user created by infra agent; `MONGO_URL` injected via `.env.platform`
- **Per-app teardown:** database dropped; container is not removed
- **Tenant teardown:** container stopped and removed when all apps in the tenant are deleted

## Documentation

Every generated app ships documentation co-located with its code in the Gitea repository. Docs are committed by the publisher in the same batch as the application files and versioned together with them.

### What is generated

| File | App types | Content |
|------|-----------|---------|
| `README.md` | All | Overview, access URL, API reference table, data model (if applicable), environment variables, usage notes |
| `docs/api.md` | `connector`, `assistant` | Expanded API reference with request/response examples and error codes |
| `docs/data-ingestion.md` | `assistant` | How to upload and index documents; supported file types; chunking strategy |

`README.md` structure:
1. **Overview** — one-paragraph description from the spec
2. **Access** — live URL at `{name}.APP_DOMAIN` and Gitea repo link
3. **API reference** — routes table (method, path, description, auth required) derived from the route manifest
4. **Data model** — entities and fields; omitted for `dashboard` type
5. **Environment variables** — table of all env vars the app reads, their purpose, and whether required
6. **Usage notes** — operator-facing instructions (e.g. how to submit a form, how to upload documents)

### How docs are surfaced

Docs are accessible in two places:
- **Gitea web UI** — directly in the repository, visible to the platform administrator
- **Portal Docs tab** — gateway proxies `README.md` and `docs/` files from Gitea (`GET /apps/{id}/docs`); portal renders markdown. Operators do not need Gitea access or credentials.

The portal is the primary reading surface for operators. Gitea is the authoritative store.

### Why co-located with code

- Docs are committed in the same batch as the application files; they are always in sync with the version that was deployed
- The Gitea repo is the single source of truth for both code and docs — no separate doc hosting, no sync process, no drift
- Remediation patches can update docs alongside code when behaviour changes

## Mandatory interface contract

Every generated app container must satisfy these contracts, enforced by `standards/patterns.yaml` and reviewed by the reviewer agent.

### Health endpoints

| Endpoint | Semantics | Required response |
|----------|-----------|-------------------|
| `GET /health` | Liveness — process is alive | `200 {"status": "ok"}` |
| `GET /ready` | Readiness — all dependencies connected | `200 {"status": "ok"}` or `503 {"status": "error"}` |

The `GET /health` endpoint is used by:
- Acceptance agent (step 1 of smoke check; retried with backoff for startup lag)
- Docker `HEALTHCHECK` directive (interval: 10s, timeout: 5s, retries: 5)

The `GET /ready` endpoint is used by:
- Acceptance agent (validates dependency connectivity)
- Load balancer routing (Traefik can be configured to gate on readiness)

### Logging

Structured JSON with required fields: `timestamp`, `level`, `service`, `message`. Secrets and tokens must never appear in log output. Promtail collects logs passively via Docker log driver — the app does not need to configure log shipping.

### Configuration

All configuration via environment variables (SCREAMING_SNAKE_CASE). No config files in the image. Secrets injected at runtime via `.env.platform` by the infra agent; the app must not hard-code any credentials.

### Graceful shutdown

10-second drain period on `SIGTERM` before exit. Rate limiting is handled by Traefik; apps must not implement their own rate limiting.

## App lifecycle

```
spec locked
    ↓
provisioning    (gateway sets status; infra agent runs if type requires resources)
    ↓
building        (planner → builder → ui-designer → reviewer → publisher; board card created)
    ↓
deploying       (Woodpecker CI: generate-tests → test → docker-build → deploy)
    ↓
accepting       (acceptance agent exercises live routes)
    ↓ pass                              ↓ fail
active                            remediation loop
    ↓ watchdog detects error       ↓ exhausted
degraded                          needs_human → operator notified with diagnosis
    ↓ remediation succeeds             ↓
active                            failed
    ↓ operator deletes
archived        (soft-delete; infra teardown runs; resources destroyed)
```

**Who drives each transition:**

| Transition | Driven by |
|-----------|-----------|
| `→ provisioning` | Gateway (`run_build` start) |
| `→ building` | Gateway (after infra provision) |
| `→ deploying` / sets `apps.repo_url` | Gateway (after publisher returns) |
| `→ active` | Gateway (after acceptance passes) |
| `→ degraded` | Gateway (on watchdog breaking error report) |
| `→ needs_human` | Gateway (on remediation exhaustion) |
| `→ failed` | Gateway (on unrecoverable build or acceptance failure) |
| `→ archived` | Gateway (on operator delete) |

## Platform components that interact with running apps

| Component | Interaction | Protocol | Timing |
|-----------|------------|----------|--------|
| **Traefik** | Routes `{name}.APP_DOMAIN` to app container via Docker label discovery | HTTP (reverse proxy) | Continuous; routes traffic for the app's entire lifetime |
| **Acceptance agent** | Exercises `GET /health`, `GET /ready`, and spec-derived routes | HTTP | One-shot at first deploy; repeated after each remediation redeploy |
| **Watchdog** | Polls logs and container health state | Docker socket (read-only) | Continuous; every `WATCHDOG_POLL_INTERVAL` seconds |
| **Woodpecker Agent** | Builds image, runs container with labels and env vars | Docker | At deploy step; once per CI pipeline run |
| **Infra agent** | Provisions Postgres DB and pgvector container; tears down on delete | Docker SDK + SQL | At build time (provision) and on delete (teardown) |
| **Promtail** | Collects stdout/stderr from all containers | Docker log driver | Continuous; passive |
| **Remediation agent** | Fetches source from Gitea, patches, redeploys via publisher | Gitea HTTP + pipeline | On failure detection |

**Key distinction — watchdog vs. acceptance:**
- Acceptance validates the app at the HTTP level using the app's public Traefik URL.
- The watchdog monitors the app at the Docker infrastructure level (logs, container state) via the Docker socket. It does not make HTTP requests to the app.

## Tenancy and isolation

Apps are scoped to a **tenant**, not directly to a user. In v1 each operator has exactly one `single_user` tenant, so the experience is identical to per-user ownership. See `docs/architecture/decisions/0003-tenant-model.md` and `docs/architecture/data-model.md` for the tenant schema.

- **Source isolation:** one Gitea repo per app, under the tenant's private Gitea org (named `user-{telegram_id}` in v1). No cross-tenant repo access.
- **Runtime tenancy marker:** `platform.tenant={tenant_id}` Docker label. The watchdog uses this to scope container sweeps to the correct tenant.
- **Network:** all apps share `platform_platform`. Apps can resolve each other by container name. There is no network-level isolation between apps from different tenants.
- **Compute:** no CPU or memory quotas enforced by the platform at this stage (future: Docker resource limits per ADR-0001 scope expansion).
- **Storage:** per-app SQLite is on the container filesystem (lost on container removal). Per-app Postgres DBs are on the shared Postgres instance; large datasets can affect platform DB performance.
- **Future team tenants:** adding a second user to a tenant requires only a `tenant_memberships` join table — no changes to this app contract, Docker labels, Gitea repo structure, or any agent code.

## Deletion and teardown sequence

1. Operator triggers delete via portal or Telegram
2. Gateway sets `apps.archived = true`
3. Gateway calls `infra POST /teardown`:
   - Drops per-app Postgres DB and user (if provisioned)
   - Stops and removes pgvector container (if provisioned)
   - Deletes `.env.platform` from Gitea repo
4. Gateway archives the Gitea repo (soft-delete; source is preserved, not destroyed)
5. App container is stopped via `docker stop {app-name}`; image is not removed (cached for potential future builds)

**Data loss warning:** per-app Postgres data is irreversibly dropped on teardown. There is no backup mechanism for app-tier databases.

## Known limitations

- No per-app compute or memory isolation; a runaway app can saturate the Docker host.
- No storage quota; a large app database can degrade platform Postgres performance.
- SQLite-backed apps (`form`, `workflow`, `dashboard`) lose all data if the container is stopped and restarted without a persistent Docker volume — the current `.woodpecker.yml` template does not mount a volume.
- No secret rotation; credentials injected by the infra agent at provision time are static for the app's lifetime.
- No rollback mechanism; remediation always produces a new commit moving the repo forward.
- pgvector containers are not monitored independently by the watchdog — a pgvector crash may generate an alert attributed to the app container rather than its resource dependency.
- The platform network membership means apps can reach each other by container name; there is no egress filtering to prevent inter-app communication.
