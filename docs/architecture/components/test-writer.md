## Purpose

The test-writer agent generates pytest test files for a scaffolded application. It runs as the first step of the Woodpecker CI pipeline (`generate-tests`), not as a gateway-dispatched service. The generated tests are executed by the subsequent `test` CI step; a failing test blocks deployment.

## Responsibilities

**Owns:**
- Fetching app source files from Gitea
- LLM-based pytest generation from source, guided by the test targets in the build plan
- Auto-correction of known LLM import mistakes
- Writing test files into the CI workspace

**Does not own:**
- Running the tests (the subsequent `test` CI step does this with `pytest`)
- Code generation (owned by the builder agent)
- CI pipeline configuration (generated by the publisher in `.woodpecker.yml`)

## Internal structure

| Component | Responsibility |
|-----------|----------------|
| `main.py` | FastAPI app, `POST /generate` endpoint |
| LangChain / Ollama | LLM calls for test generation; receives source + test targets from build plan |
| Import correction | Post-processes LLM output to fix known bad imports (e.g. wrong `TestClient` path) |

## Key flows

### Woodpecker CI step execution

1. Woodpecker CI runs the `generate-tests` step, which calls `POST /generate` with `{repo, user_id}`
2. Test-writer fetches source files from Gitea via HTTP API
3. Sends source, standards, and test targets to Ollama; LLM returns pytest file content
4. Auto-corrects known import mistakes in LLM output
5. Writes test files to the CI workspace path
6. Returns `{files_written: [...]}` to the CI step
7. The next CI step (`test`) runs `pip install -r requirements.txt && pytest`
8. Failing pytest blocks the `docker-build` and `deploy` steps

### Fallback: LLM output unparseable

1. LLM returns output that cannot be identified as valid Python
2. Test-writer writes a minimal health-check test (imports the app, calls `/health`, asserts 200)
3. Build proceeds with the fallback test

## Data owned

The test-writer agent has no direct database access.

**Writes (filesystem only):**
- Test files are written to the Woodpecker CI workspace path (`/workspace/tests/`); not committed to Gitea by the test-writer. The CI pipeline runs them in place.

**Reads:**
- App source files — fetched from Gitea via HTTP API at CI time
- Standards YAML files — loaded from filesystem at startup

## Error handling and failure modes

| Failure | Behaviour |
|---------|-----------|
| Gitea unreachable | CI step fails; Woodpecker marks the `generate-tests` step as failed; pipeline does not proceed to `test` |
| Ollama unreachable | Falls back to the minimal health-check test; CI step succeeds with fallback |
| LLM returns invalid Python | Import auto-corrector runs; if still invalid, falls back to health-check test |
| CI workspace path not writable | CI step fails; pipeline aborted |

## Non-functional constraints

- Called exclusively as a Woodpecker CI step; not exposed on the platform Docker network for general use.
- Fallback test is minimal — it validates the container starts and `/health` returns 200, but not any functional behaviour.
- Test generation latency adds to total CI pipeline time; typically 10–30 s.

## External interfaces

### Exposes

| Endpoint | Caller | Purpose |
|----------|--------|---------|
| `POST /generate` | Woodpecker CI step | Generate and write test files |
| `GET /health` | Woodpecker / Traefik | Health check |

### Calls

| Target | Purpose |
|--------|---------|
| Gitea HTTP API | Fetch app source files |
| Ollama HTTP API | LLM test generation |

## Configuration

| Variable | Default | Notes |
|----------|---------|-------|
| `TEST_WRITER_LLM_PROVIDER` | `ollama` | LLM provider: `ollama`, `openai`, `anthropic` |
| `TEST_WRITER_LLM_MODEL` | provider default | Model name. Defaults: `llama3.1:8b` (ollama), `gpt-4o` (openai), `claude-sonnet-4-6` (anthropic) |
| `TEST_WRITER_LLM_API_KEY` | — | API key for the chosen provider. Falls back to `OPENAI_API_KEY` / `ANTHROPIC_API_KEY`. Required when provider ≠ `ollama` |
| `OLLAMA_URL` | `http://ollama:11434` | Used only when `TEST_WRITER_LLM_PROVIDER=ollama` |
| `GITEA_URL` | — | Internal Gitea base URL |
| `GITEA_ADMIN_USER` / `GITEA_ADMIN_PASS` | — | Gitea API credentials for fetching source |

## Known limitations

- Tests are written once at CI time — there is no mechanism to regenerate them if the source changes on a subsequent push.
- The import auto-correction list is hardcoded; new LLM model versions may introduce different mistake patterns.
- Fallback health-check test is minimal and may pass even when the application is functionally broken.
