## Purpose

The template library is the set of base scaffolds that the builder and ui-designer use as starting points for every generated application. It is not a running service — it is a versioned directory tree mounted read-only into both agents at container startup. Templates guarantee the mandatory platform contract (health endpoints, Dockerfile security, CI pipeline structure) without relying on LLM output for boilerplate. The LLM's role is to add app-specific code on top of the template, not to regenerate infrastructure concerns from scratch.

## Responsibilities

**Owns:**
- Base scaffolds for all supported API stacks (FastAPI, Express)
- Base frontend scaffold (MUI Minimal Next.js 15)
- All mandatory files that the generated app contract requires (Dockerfile, health endpoints, structured logging, `.woodpecker.yml` skeleton, env var wiring)
- Versioning: templates are tagged; planner build plans reference a template version to ensure reproducible builds

**Does not own:**
- App-specific routes, models, and business logic (generated by the builder via LLM, layered on top of the template)
- UI pages, forms, and data-bound components (generated by the ui-designer via LLM, layered on top of the frontend template)
- CI pipeline completion (`.woodpecker.yml` template has the structure; builder fills in app-specific steps)

## Directory structure

Templates are stored in `templates/` at the root of this repository and mounted read-only into the builder at `/templates` and into the ui-designer at `/templates`.

```
templates/
  api/
    fastapi-base/          # Python FastAPI API template
      main.py              # FastAPI app with /health, /ready, structured logging, lifespan
      config.py            # Pydantic Settings — reads all config from env vars
      logging.py           # Structured JSON logger (timestamp, level, service, message)
      requirements.txt     # Pinned: fastapi, uvicorn, pydantic-settings, structlog
      Dockerfile           # Multi-stage; python:3.12-slim; non-root USER appuser; no :latest
      .woodpecker.yml      # generate-tests → test → docker-build → deploy (API container)
      README.md            # Template placeholder — builder replaces with generated docs

    express-base/          # Node.js Express API template
      index.js             # Express app with /health, /ready, structured logging
      config.js            # Config from process.env; no defaults for secrets
      logger.js            # Structured JSON logger (pino)
      package.json         # Pinned: express, pino, dotenv; Node 22 engines field
      Dockerfile           # Multi-stage; node:22-alpine; non-root USER node; no :latest
      .woodpecker.yml      # generate-tests → test → docker-build → deploy (API container)
      README.md            # Template placeholder

  frontend/
    mui-minimal/           # Next.js 15 + MUI Minimal v7 frontend template
      src/
        app/
          layout.tsx       # Root layout with MUI ThemeProvider, fonts, metadata
          page.tsx         # Landing page placeholder — ui-designer replaces content
        theme/             # MUI theme: palette, typography, component overrides
        lib/
          api.ts           # Axios instance; base URL from NEXT_PUBLIC_API_URL
      public/              # Static assets (logo placeholder)
      package.json         # Pinned: next, @mui/material, @mui/icons-material, axios, swr
      Dockerfile           # node:22-alpine; standalone output; non-root USER node
      next.config.mjs      # output: 'standalone'; rewrites for API proxy
      .env.example         # Documents NEXT_PUBLIC_API_URL
```

## Template contract

Every template guarantees the following platform requirements without LLM involvement:

### API templates (fastapi-base, express-base)

| Requirement | Implementation |
|-------------|----------------|
| `GET /health` returns `200 {"status": "ok"}` | Hardcoded in `main.py` / `index.js` |
| `GET /ready` returns `200 {"status": "ok"}` or `503` | Hardcoded; subclasses add dependency checks |
| Structured JSON logging (`timestamp`, `level`, `service`, `message`) | `logging.py` / `logger.js`; injected at app startup |
| All configuration via env vars (no hardcoded secrets) | `config.py` (Pydantic Settings) / `config.js` (process.env) |
| Non-root Docker user | `USER appuser` / `USER node` in Dockerfile |
| Pinned base image (no `:latest`) | Enforced in Dockerfile; reviewer checks |
| 10-second graceful shutdown on `SIGTERM` | Uvicorn `--timeout-graceful-shutdown 10` / Express SIGTERM handler |
| `.woodpecker.yml` with `generate-tests → test → docker-build → deploy` steps | Template `.woodpecker.yml` |

### Frontend template (mui-minimal)

| Requirement | Implementation |
|-------------|----------------|
| MUI Minimal v7 design system | `src/theme/` from MUI Minimal template; palette and component overrides |
| API connectivity via gateway env var | `NEXT_PUBLIC_API_URL` in `lib/api.ts` |
| Next.js standalone output for Docker | `output: 'standalone'` in `next.config.mjs` |
| Non-root Docker user | `USER node` in Dockerfile |
| No `:latest` base image | Pinned in Dockerfile |

## How the builder uses templates

1. Planner build plan includes `{"stack": "fastapi", "frontend": true, "template_version": "v1"}`
2. Builder reads the template for the selected stack from `/templates/api/fastapi-base/`
3. Builder sends template files + spec + build plan to Ollama with the instruction: *"These template files implement the platform contract. Add app-specific routes, models, and business logic. Do not modify health endpoints, logging setup, or Dockerfile USER directives."*
4. LLM returns only the files it needs to add or modify; template files that require no change are passed through unchanged
5. Builder merges LLM output into the template file set; returns the complete file set to the gateway

## How the ui-designer uses the frontend template

1. UI designer receives: spec, route manifest, data model, stack selection (`frontend: true`)
2. UI designer reads the frontend template from `/templates/frontend/mui-minimal/`
3. UI designer sends template structure + spec + routes to Ollama: *"Using the MUI Minimal template as a base, generate Next.js pages and components for the following routes and data model. Keep the theme, layout, and API wiring patterns from the template."*
4. LLM returns: `src/app/` pages, `src/sections/` components, any additional MUI component usage
5. UI designer merges LLM output with the base template

## File layout in the generated Gitea repo

Apps with a frontend produce a two-subdirectory structure:

```
{app-name}/
  api/                    # API backend (from fastapi-base or express-base template + LLM)
    main.py
    Dockerfile
    requirements.txt
    .woodpecker.yml        # Builds and deploys both api/ and frontend/ containers
  frontend/               # Frontend (from mui-minimal template + LLM)
    src/
    Dockerfile
    package.json
  README.md               # Generated by builder docs.py; one file for the whole app
  docs/
    api.md                # connector and assistant types only
    data-ingestion.md     # assistant type only
```

Apps without a frontend (connector):

```
{app-name}/
  main.py                 # API files at root (no subdirectory)
  Dockerfile
  requirements.txt
  .woodpecker.yml
  README.md
  docs/
    api.md
```

The Traefik routing labels for the frontend container use `{app-name}-ui.APP_DOMAIN`; the API container uses `{app-name}.APP_DOMAIN` (unchanged).

## Versioning and updates

Templates are versioned by the directory tag in the planner build plan. The planner always selects the current version from `standards/templates.yaml`. To update a template:

1. Apply changes to the `templates/` directory
2. Bump `current_version` in `standards/templates.yaml`
3. Rebuild the builder and ui-designer Docker images (templates are baked in; no runtime hot-reload)
4. New builds use the new template; existing deployed apps are unaffected

Security patches to a template (e.g. bumping a pinned dependency) do **not** automatically redeploy existing apps. A remediation job or manual rebuild is required for in-flight apps.

## Data owned

The template library is a static artifact — it owns no data and writes no state. Templates are read-only at runtime.

## External interfaces

None. Templates are consumed by the builder and ui-designer via filesystem reads, not HTTP calls.

## Non-functional constraints

- Templates are baked into the builder and ui-designer Docker images at build time. No runtime filesystem writes or hot-reload.
- Template files must remain small enough to fit within the LLM context window alongside the spec and build plan. Large template files should be split or summarised in the prompt rather than included verbatim.
- The reviewer does not re-check template-provided files (Dockerfile, logging, health endpoints). It only reviews LLM-generated additions. This is enforced by passing template file paths to the reviewer as an exclusion list.

## Known limitations

- Only two API stacks (FastAPI, Express) and one frontend template (MUI Minimal) are supported. Other stacks require a new template and a planner update.
- Updating a template dependency does not automatically redeploy existing apps — operators must manually trigger a rebuild or wait for the next remediation cycle.
- LLM augmentation may accidentally modify template-provided files (health endpoints, Dockerfile USER directive). The reviewer's static checks catch these regressions.
- The frontend template assumes English-language content. Internationalisation (i18n) is out of scope for v1.
